Classroom Token Hub For District Partners

Last Updated: April 15, 2026

Overview

Thank you for your interest in Classroom Token Hub (CTH).

Classroom Token Hub is a teacher-developed classroom economy platform designed around data minimization, class-scoped isolation, and limited retention of student-identifying information.

This document summarizes the current v1 architecture for district review. It is intended to describe how the live application handles classroom, student, and operational data at a high level.

For implementation details and source review, please refer to the official GitHub repository.

1. Data Collection, Processing, and Storage

Classroom Token Hub is designed to collect only the data needed to create classroom rosters, let students claim the correct account, and operate the classroom economy. Sensitive fields are hashed or encrypted before storage, and student-identifying claim data is reduced after setup.

When a teacher creates student roster seats, the system stores the following classroom-scoped information:

  • Student first name: Used for teacher roster display and in-app greeting
  • Student last initial: Used for roster display
  • Student last name matching hashes: Used only during claim and recovery matching
  • Date-of-birth derived integer (`dob_sum`): Used only during initial claim matching
  • Class block / class label: Used to organize the teacher's roster and class context
  • Join code: Used to bind a student to the correct class context during claim and recovery

The v1 claim flow processes that information as follows:

  • First name is stored encrypted for roster display and student greeting
  • Last initial is stored in readable form for display because it is part of the student-facing and teacher-facing roster UI
  • Last name is not retained in readable form for normal operation; the system stores one-way matching hashes for claim and recovery checks
  • Date of birth is not stored as a raw date; the app derives a non-reversible integer value used during initial claim, then removes that claim-specific value after setup
  • Students claim accounts using join code + first initial + last name + date of birth, after which they create a username, PIN, and passphrase for ongoing access

NOTE

In v1, the core student setup flow does not rely on permanent storage of full last names or raw birth dates.

Claim-specific identifiers are reduced after account setup so ongoing authentication depends on the student's chosen credentials rather than retained roster PII.

2. Class-Level Data Isolation

Each class operates within a class-scoped context anchored by a join code and teacher ownership.

  • Teachers can access only the classes they own
  • Roster seats, transactions, attendance, settings, and balances are scoped to class context
  • System design avoids a district-wide student directory for normal application use
  • Identical student names in different classes are not treated as a shared global identity by default

3. Student Claim and Recovery Model

Students do not sign in with teacher names, email addresses, or district-wide accounts.

  • Initial claim is class-scoped through the class join code
  • After claim, students use a username, PIN, and passphrase
  • Teacher-issued recovery uses class-scoped recovery controls rather than persistent storage of full roster PII

4. Teacher and Student Data Retention

The application distinguishes between classroom-state data needed to operate the economy and temporary claim data that is reduced after setup.

  • Claim-only matching data is removed or reduced after successful account setup
  • Classroom-state data remains while the class is active
  • Deleting a class removes data associated with that class context
  • If a student no longer belongs to any remaining class, the student account is also deleted
  • The app does not provide a restorable classroom-state backup feature for deleted classes

5. Operational Logging and Technical Retention

To operate and debug the service, the app keeps limited technical records separate from classroom content.

  • Request traces are retained for about 7 days
  • Structured error events are retained for about 14 days
  • Database-backed error log entries are retained for about 14 days
  • Application log files are rotation-based and size-bounded
  • These records are intended for debugging, support correlation, and service reliability rather than classroom monitoring

6. Administrative Access Model

CTH does not provide a broad district-style console for browsing student roster data across all classes.

  • Teachers manage only their own classes
  • System administrators operate with a limited technical support scope
  • Normal administrator workflows emphasize usernames, class labels, counts, and technical identifiers rather than full student roster identity data

7. Application vs Infrastructure Access

Application-level interfaces are intentionally narrower than raw infrastructure access. Production infrastructure access is reserved for system maintenance, debugging, and incident response, and is not designed as a classroom-observation tool.

8. Security

  • HTTPS/TLS encryption in transit
  • Encryption at rest for sensitive stored fields
  • Strong one-way hashing for credentials and claim-verification values
  • Class-scoped access controls enforced at the application layer
  • Timed confirmation gates for destructive actions such as deleting class data
  • Reduced retention of claim-specific student identity data after setup
  • Restricted operational access for maintenance and incident handling

9. Compliance and Standards

FERPA

Classroom Token Hub is designed to align with FERPA-oriented data minimization principles by limiting stored student identity data, restricting classroom access to authorized educators, and avoiding third-party advertising or analytics use.

Accessibility (WCAG)

Classroom Token Hub is developed with accessibility in mind and aims toward WCAG 2.1 Level AA practices where feasible. The interface is continuously improved to support a wide range of users, including those using assistive technologies.

While formal accessibility audits have not yet been conducted, the system is developed with accessibility best practices as a core consideration. Districts with specific compliance requirements may choose to conduct independent evaluations.

10. What CTH Does Not Do

  • No advertising or data selling
  • No third-party analytics
  • No behavioral profiling
  • No cross-class tracking
  • No required integration with external identity systems such as SSO or LTI in the hosted v1 app
  • No restorable deleted-class archive in normal operation

NOTE

Classroom Token Hub is designed to operate as a complete, self-contained system without external integrations such as SSO or LTI.

District partners may choose to create a self-hosted fork of the project to meet specific infrastructure or compliance requirements. In such cases, the forked system operates independently, may be renamed, and is no longer governed by, nor representative of, the original system’s design, guarantees, or invariants.
