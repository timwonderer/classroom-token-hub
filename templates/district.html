Classroom Token Hub For District Partners

Last Updated: April 15, 2026

Overview

Thank you for your interest in Classroom Token Hub (CTH).

Classroom Token Hub is a teacher-developed classroom management system with a simulated economic model, designed with a data-minimization and isolation-first architecture.

As a 100% teacher-designed, developed, hosted, and maintained platform, we look forward to working with district partners to bring CTH to teachers and students.

This document is designed to provide an overview of the system's architecture and data protection measures for district review. The content of this document represents the most up-to-date information regarding the system's design and implementation. For more details, please visit our official GitHub repository.

1. Data Collection, Processing, and Storage

Classroom Token Hub strives to collect only the minimum data necessary to operate effectively while maintaining a strong teacher and student user experience. Sensitive user-provided information is never stored as plaintext. Where storage in readable form is required for functionality (such as display labels like class sections), the data is treated as non-sensitive, remains strictly scoped to the class, and is not used for identification or cross-system linkage.

When a teacher creates a class, they will provide the following information on the roster template:

  • Student First Name: Used for teacher roster display, greetings, and account claiming process
  • Student Last Name: Used for teacher roster display, greetings, and account claiming process
  • Student Class Section: Used for creating class sections within the roster page
  • Additional Notes: Used for internal teacher notes

Our server processes the information from the roster template using the following approach:

  • Student First Name and Last Name (temporary): Normalized to remove whitespace, special characters, and accent marks. Hashed using HMAC-SHA256 with a per-record salt and a global pepper before storage. Use as account look up during claim. Removed upon successful claim and account creation.
  • Student First Name and Last Name (persistent): Encrypted individually with whitespace, special character, and accent marks preserved before storage. Use as in-app greeting and roster display for teachers.
  • Student Class Section: Stored unchanged as a non-sensitive, teacher-defined label and not interpreted, parsed, or used for inference by the system.
  • Additional Notes: Encrypted unchanged and stored. Used for teacher internal records

NOTE

Student Class Section and Additional Notes are treated as unstructured text within the application and are used for display only.

These fields are not interpreted, parsed, or used for identification, inference, or external context derivation by the system.

2. Class-Level Data Isolation

Each class operates as an independent system defined by a unique class identifier.

  • No cross-class visibility or aggregation
  • Teachers can only access classes they create
  • Identical names across classes are treated as unrelated
  • The system does not maintain a global student identity across classes

3. Name Collision Handling

When duplicate names exist within a class, a random deduplication code is used during claim.

  • Temporary and class-scoped
  • No persistent identifiers introduced

4. Optional Teacher Notes

Teachers may add notes for internal use.

  • Stored as encrypted, unstructured text
  • Not parsed or analyzed by the system

5. Data Lifecycle & Deletion

  • Class deletion removes all associated data
  • Teacher deletion removes all classes and student data
  • 180-day inactivity triggers automatic deletion
  • No archival or soft-delete exists

6. Administrative Access Model

CTH does not provide administrators with a general interface to browse or query data.

  • Access is teacher-initiated and class-scoped
  • Users are represented by pseudonymous identifiers
  • No cross-class inspection or bulk access tools exist

7. Application vs Infrastructure Access

Application-level tools do not expose identifiable data. Infrastructure access is restricted to system maintenance and not used for monitoring user activity.

8. Security

  • HTTPS encryption in transit
  • Encrypted sensitive data at rest
  • Strict class-scoped access controls enforced at the application level
  • No administrative interface for browsing or querying student data
  • Pseudonymous identifiers used in place of direct identity references
  • System architecture prevents cross-class data access by design
  • Restricted and audited infrastructure access for maintenance purposes only
  • No global admin identity or privileges or superuser capabilties by design

9. Compliance and Standards

FERPA

Classroom Token Hub is designed to operate in alignment with FERPA principles. The system minimizes the collection of personally identifiable information, restricts access to authorized educators within a class context, and does not share student data with third parties.

COPPA (School Consent Model)

Classroom Token Hub complies with COPPA through the School Consent Model. Because CTH is designed strictly for classroom educational use under the supervision of a teacher or school district, the school/educator acts as the consenting authority. Schools authorize CTH to collect and process student first names and last initials for the sole benefit of the educational program, in lieu of direct parental consent.

As a key defense of student privacy, the system does not collect or store any external identifiers (such as student email addresses, phone numbers, or national student IDs). Accounts are self-contained and purely local to the specific class period. Usernames are pseudonymously generated using a theme word and student initials, and student names are encrypted at rest with only the first name and last initial displayed. Since no external identifiers or parental contact details are collected, the system provides a high level of isolation and cannot be correlated with external databases or tracked across other platforms.

Accessibility (WCAG)

Classroom Token Hub is designed with accessibility in mind and follows WCAG 2.1 Level AA guidelines where feasible. The platform is continuously improved to support a wide range of users, including those using assistive technologies.

While formal accessibility audits have not yet been conducted, the system is developed with accessibility best practices as a core consideration. Districts with specific compliance requirements may choose to conduct independent evaluations.

10. What CTH Does Not Do

  • No advertising or data selling
  • No third-party analytics
  • No behavioral profiling
  • No cross-class tracking
  • No integration with external identity systems (e.g., SSO)
  • No archival or long-term data retention beyond active class lifecycle

NOTE

Classroom Token Hub is designed to operate as a complete, self-contained system without external integrations such as SSO or LTI.

District partners may choose to create a self-hosted fork of the project to meet specific infrastructure or compliance requirements. In such cases, the forked system operates independently, may be renamed, and is no longer governed by, nor representative of, the original system’s design, guarantees, or invariants.
