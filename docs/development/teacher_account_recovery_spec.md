# Teacher Account Recovery — Technical Specification

> **Reverse-engineered from implementation.**
> Every claim in this document has been verified against the source code in
> `app/routes/admin.py`, `app/routes/system_admin.py`, and `app/routes/student.py`
> (and `app/models.py` for the data model).

---

## Purpose

This document specifies the architecture, data model, security constraints, and
invariants for the **teacher account recovery** feature.

The goal is to allow a teacher who has lost access to their TOTP authenticator
(and therefore cannot log in) to reset their username and TOTP secret without
requiring email, phone verification, or system-administrator intervention.

Recovery is verified via two independent factors:

1. **Something you know** — Date of birth (verified as a non-reversible HMAC hash).
2. **Something you have** — Passphrase-verified 6-digit codes generated by
   students across all of the teacher's active class periods.

---

## Scope

This spec covers **teacher** credential recovery only. Student account recovery
(teacher-initiated reset of a student's credentials) is a separate feature
specified in `account_recovery_and_reclaim_spec.md`.

---

## Data Model

### `admins` table (recovery-relevant columns)

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `username` | String(80) | No | Unique; reset by recovery |
| `totp_secret` | String(200) | No | Base64-encoded encrypted secret; reset by recovery |
| `dob_sum_hash` | String(64) | Yes | HMAC-SHA256 of `str(month + day + year)`, salted. `NULL` until setup-recovery or signup. |
| `salt` | LargeBinary(16) | Yes | Random salt for `dob_sum_hash`. Generated at signup or setup-recovery. |

### `recovery_requests` table

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | Integer PK | No | |
| `admin_id` | FK → admins | No | Indexed |
| `dob_sum_hash` | String(64) | Yes | Copy of teacher's hash at time of request (audit trail) |
| `status` | Enum | No | `pending` / `verified` / `expired` / `cancelled` |
| `created_at` | DateTime(tz) | No | UTC |
| `expires_at` | DateTime(tz) | No | Set to `created_at + 5 days` |
| `completed_at` | DateTime(tz) | Yes | Set when status → `verified` |
| `partial_codes` | JSON | Yes | Array of partially-entered codes (save-progress feature) |
| `resume_pin_hash` | String(64) | Yes | HMAC of 6-digit resume PIN (no salt) |
| `resume_new_username` | String(100) | Yes | Temporarily stores desired new username during partial save |

### `student_recovery_codes` table

| Column | Type | Nullable | Notes |
|---|---|---|---|
| `id` | Integer PK | No | |
| `recovery_request_id` | FK → recovery_requests | No | Indexed |
| `student_id` | FK → students | No | Indexed |
| `code_hash` | String(64) | Yes | HMAC of 6-digit student-generated code (no salt). `NULL` until student verifies. |
| `verified_at` | DateTime(tz) | Yes | Set when student generates their code |
| `notified_at` | DateTime(tz) | No | Defaults to `utc_now()` at row creation |
| `dismissed` | Boolean | No | Student dismissed the notification banner; defaults `False` |

## DOB Sum Algorithm

The DOB sum is a non-reversible integer derived from a calendar date:

```
dob_sum = date.month + date.day + date.year
```

Example: `1990-07-15` → `7 + 15 + 1990 = 2012`

The sum is then HMAC-hashed with a per-teacher random 16-byte salt:

```python
dob_sum_hash = HMAC-SHA256(str(dob_sum).encode(), salt)
```

Accepted input formats: `YYYY-MM-DD` (primary), `MM/DD/YYYY` (fallback).

---

## Prerequisite — DOB Registration

Before self-service recovery can be initiated, the teacher's `dob_sum_hash`
must be set. There are two code paths that set it:

1. **At signup** (`POST /admin/signup`): DOB is collected during the TOTP
   confirmation step. A new random salt is generated; the HMAC hash is stored.
   The account is created with `dob_sum_hash` already populated.

2. **Via setup-recovery** (`POST /admin/setup-recovery`): Required for accounts
   created before this feature existed. Triggered by the "Setup Account Recovery"
   banner shown on the dashboard when `admin.dob_sum_hash is None`. Requires an
   active teacher session (`@admin_required`). A new random salt is generated;
   the HMAC hash is stored.

If `dob_sum_hash` is `NULL`, the self-service flow rejects the request with:
`"Teacher account not configured for recovery."`

## Self-Service Recovery Flow

### Step 1 — Initiate Recovery Request (`POST /admin/recover`)

**Rate limit:** 5 per hour (per Flask-Limiter).

The teacher submits:
- `student_usernames`: comma-separated list of student usernames (one per active class period)
- `dob_sum`: date of birth (date picker input, `YYYY-MM-DD` format)

**Backend validation sequence:**
1. Parse `dob_sum` input → compute `month + day + year`.
2. Look up each username via `username_lookup_hash` (case-insensitive HMAC of
   lowercased username). Falls back to exact `username` match for legacy data.
3. Fail with `"No matching students found."` if none resolve.
4. Compute the intersection of `admin_id` sets from `student_teachers` for all
   found students. Fail with `"do not all belong to the same teacher"` if the
   intersection is not exactly one teacher.
5. Load the teacher record. Fail if `dob_sum_hash` is `NULL`.
6. Recompute `HMAC(str(dob_sum), teacher.salt)` and compare with
   `teacher.dob_sum_hash`. Fail with `"Unable to verify your identity."` on mismatch.
7. Compute `teacher_blocks`: the set of distinct `block` values across all
   students linked to this teacher.
8. Compute `selected_blocks`: the set of `block` values from the submitted students.
9. Fail if `selected_blocks` does not cover all of `teacher_blocks`
   (missing-period check).
10. Fail if `len(students) < len(teacher_blocks)`.
11. Check for an existing `pending` `RecoveryRequest` with `expires_at > now`.
    If one exists, redirect to recovery-status (no duplicate request is created).

**On success:**
- Create `RecoveryRequest` with `status='pending'`, `expires_at = now + 5 days`.
- Create one `StudentRecoveryCode` row per selected student (code_hash = NULL).
- Store `recovery_request_id` in the session.
- Redirect to `/admin/recovery-status`.

---

### Step 2 — Student Verification (`POST /student/verify-recovery/<code_id>`)

Each selected student must log in to their own account and navigate to the
recovery notification shown on their dashboard. The notification is suppressed
once `dismissed=True` or `code_hash IS NOT NULL`.

**Query for the notification banner:**
```sql
SELECT * FROM student_recovery_codes
JOIN recovery_requests ON ...
WHERE student_id = :student_id
  AND dismissed = FALSE
  AND code_hash IS NULL
  AND status = 'pending'
  AND expires_at > :now
LIMIT 1
```

**On the verification page (`GET /student/verify-recovery/<code_id>`):**
- Validates that the `StudentRecoveryCode` belongs to the logged-in student.
- Validates that `code_hash IS NULL` (not already verified).
- Validates that the parent `RecoveryRequest.expires_at > now`.

**On submission (`POST /student/verify-recovery/<code_id>`):**
1. Verify the student's `passphrase` against `student.passphrase_hash` using
   `werkzeug.security.check_password_hash`.
2. On success: generate a 6-digit numeric code using
   `secrets.randbelow(10)` for each digit.
3. Compute `HMAC(code.encode(), b'')` (no salt) and store in `code_hash`.
4. Set `verified_at = utc_now()`.
5. Display the 6-digit code on screen for the student to give to the teacher.

Students may also dismiss the notification without verifying (`dismissed = True`),
but this does not invalidate the request.

---

### Step 3 — Enter Recovery Codes and New Username (`POST /admin/reset-credentials`)

**Rate limit:** 10 per hour.

The teacher enters the codes collected from each student and their desired new
username.

**Validation sequence:**
1. Session must contain `recovery_request_id`.
2. `RecoveryRequest.status` must be `'pending'`.
3. All `StudentRecoveryCode` rows must have `code_hash IS NOT NULL`
   (all students must have verified before this step proceeds).
4. Count of entered codes must equal count of `StudentRecoveryCode` rows.
   **On mismatch:** all codes are immediately invalidated
   (`code_hash = NULL`, `verified_at = NULL`) and the teacher is redirected
   back to the status page.
5. Each entered code must be exactly 6 decimal digits. **On invalid format:**
   all codes are immediately invalidated.
6. Compute `HMAC(code.encode(), b'')` for each entered code, build a set.
   Compare against the stored `code_hash` set (order-independent comparison).
   **On mismatch:** all codes are immediately invalidated.
7. Validate that `new_username` is not already in use by a different teacher.

**On success:**
- Generate a new TOTP secret with `pyotp.random_base32()`.
- Compute the provisioning URI (issuer: `"Classroom Economy Admin"`).
- Render a QR code as a base64-encoded PNG.
- Store `totp_secret` and `new_username` in the session (not yet committed to DB).
- Display the QR code for the teacher to scan.

**Code invalidation function** (`_invalidate_all_recovery_codes`):
- Sets `code_hash = None` and `verified_at = None` on all rows.
- Commits immediately.
- Logs the invalidation count.
- Forces students to regenerate codes before another attempt.

---

### Step 4 — Confirm TOTP and Complete Reset (`POST /admin/confirm-reset`)

**Rate limit:** 10 per hour.

The teacher scans the QR code and submits the first 6-digit TOTP code from
their authenticator app.

**Validation:**
- Session must contain `reset_totp_secret` and `reset_new_username`.
- Verify the submitted TOTP code against the new secret using `pyotp.TOTP.verify()`.

**On success:**
- `admin.username = new_username`
- `admin.totp_secret = encrypt_totp(totp_secret)` (AES-encrypted before storage)
- `recovery_request.status = 'verified'`
- `recovery_request.completed_at = utc_now()`
- Clear `reset_totp_secret` and `reset_new_username` from session.
- Redirect to `/admin/login`.

## Partial-Progress Save / Resume

If the teacher cannot collect all student codes in one session, they can save
partial progress and return later.

### Save (`POST /admin/save-recovery-progress`)

**Rate limit:** 10 per hour.

- Accepts partial list of entered codes and a desired new username.
- Generates a 6-digit resume PIN using `secrets.randbelow(10)` for each digit.
- Stores `HMAC(resume_pin.encode(), b'')` (no salt) in `recovery_request.resume_pin_hash`.
- Stores entered codes in `recovery_request.partial_codes` (JSON).
- Stores the new username in `recovery_request.resume_new_username`.
- Displays the plaintext PIN once — teacher must record it.

### Resume (`POST /admin/resume-credentials`)

**Rate limit:** 10 per hour.

- Teacher enters their 6-digit resume PIN.
- Computes `HMAC(pin.encode(), b'')` and looks up the matching `pending`,
  non-expired `RecoveryRequest`.
- On match: sets `recovery_request_id` and `resume_mode = True` in session.
- Redirects to `/admin/reset-credentials`, which pre-populates the form with
  `recovery_request.partial_codes` and `recovery_request.resume_new_username`.

## System Admin Override — TOTP Reset (`POST /sysadmin/admins/<id>/reset-totp`)

When self-service recovery is unavailable (e.g., `dob_sum_hash` is NULL or
recovery window has passed), a system administrator can reset the teacher's
TOTP directly.

**Requirements:** Active system admin session (`@system_admin_required`).

**Action:**
1. Generate a new TOTP secret with `pyotp.random_base32()`.
2. Encrypt with `encrypt_totp()` and store in `admin.totp_secret`.
3. Commit immediately.
4. Return JSON with `status`, `message`, encrypted secret, plaintext secret,
   base64-encoded QR code PNG, and `username`.

The teacher's username is **not changed** by this path. Only the TOTP secret
is replaced.

## Security Constraints

| Constraint | Implementation |
|---|---|
| Recovery initiation rate limit | 5 per hour |
| Reset/confirm/save/resume rate limit | 10 per hour each |
| DOB sum storage | HMAC-SHA256 with random 16-byte salt; not reversible |
| Recovery request expiry | 5 days from creation |
| Student code format | 6 decimal digits; cryptographically random (`secrets.randbelow`) |
| Student code storage | HMAC-SHA256 with empty salt; not stored in plaintext |
| Code comparison | Set comparison (order-independent) |
| Code invalidation on failure | ALL codes nulled immediately on any failed submission |
| New TOTP before commit | Stored in session only; committed only after TOTP confirmation |
| Username uniqueness | Checked before issuing QR code |
| Student scoping | Recovery lookup must resolve to exactly one common teacher |
| Period coverage | All teacher's active periods must be represented |

## State Machine — RecoveryRequest

### States

- `pending` — request created; awaiting student codes and teacher submission
- `verified` — credentials successfully reset
- `expired` — detected at `/admin/recovery-status` when `expires_at < now`
- `cancelled` — reserved; not currently set by any route

### Transitions

| From | To | Trigger |
|---|---|---|
| _(none)_ | `pending` | Teacher creates a new recovery request |
| `pending` | `verified` | Teacher submits correct codes and confirms TOTP |
| `pending` | `expired` | `/admin/recovery-status` is loaded after `expires_at` |
| `pending` | `pending` | All codes invalidated on failed code submission (request itself stays pending) |

### Constraints

- Only one active (`pending`, non-expired) `RecoveryRequest` per teacher at a time.
  A second initiation redirects to the existing request.
- `RecoveryRequest` rows are never deleted; only status-transitioned.

---

## Hard Boundaries

The teacher recovery system must NOT:

- Modify any student's economic data or transaction records.
- Alter the teacher's classes, join codes, or student roster.
- Create a new teacher account (only the existing account is updated).
- Change the teacher's `dob_sum_hash` or `salt` (these are read-only after
  setup).
- Expose which teachers exist to unauthenticated callers — all error messages
  are generic.

## Acceptance Test Cases

*(All covered by `tests/test_teacher_recovery.py` and
`tests/test_sysadmin_recovery.py`.)*

### DOB Registration

- Teacher registered via signup always has `dob_sum_hash` set.
- Legacy teacher without `dob_sum_hash` sees "Setup Account Recovery" prompt on dashboard.
- After `POST /admin/setup-recovery`, `dob_sum_hash` is non-NULL.

### Initiation Guards

- Wrong DOB → "Unable to verify your identity."
- Missing period → "must select at least one student from each of your active periods."
- Students from different teachers → rejected.
- Teacher with `dob_sum_hash = NULL` → "Teacher account not configured for recovery."
- Existing active request → redirected to status page (no duplicate).
- Username lookup is case-insensitive (uses `username_lookup_hash`).

### Code Verification

- Recovery request expiry is checked before student can verify.
- Double-verification is rejected ("already verified").
- Passphrase mismatch on student side → code is not generated.

### Reset Validation

- Submission with not-all-students-verified → redirect to status.
- Wrong count of codes → all codes invalidated.
- Invalid code format (non-numeric or not 6 digits) → all codes invalidated.
- Mismatched codes → all codes invalidated.
- Duplicate new username → error without invalidating codes.

### Completion

- After `confirm-reset`, `admin.username` equals the submitted new username.
- After `confirm-reset`, `admin.totp_secret` is different from the original.
- `recovery_request.status = 'verified'`.
- `recovery_request.completed_at` is set.
- No student records, transactions, or balances are modified.

### Sysadmin Override

- `POST /sysadmin/admins/<id>/reset-totp` returns JSON with new QR code.
- Unauthenticated call returns HTTP 302 to login.
- Teacher can log in with new TOTP secret after sysadmin reset.
