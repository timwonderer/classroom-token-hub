## Stage 2 – Economic Invariant Findings (Full Validation)

### Finding 1: Multi-tenant Balance Leakage (Global Balance Calculation)

**Audit Claim:** `Student.checking_balance` and `Student.savings_balance` properties (`app/models.py:290-299`) sum transactions by `student_id` only, with no `join_code` filter. Funds earned in one class economy would be visible/spendable in another.

**Verdict: PARTIALLY CONFIRMED**

**What is true:**
- The simple properties at `app/models.py:290-299` do aggregate across all join codes:
  ```python
  @property
  def checking_balance(self):
      total = sum((_quantize_currency(tx.amount) for tx in self.transactions
                   if tx.account_type == 'checking' and not tx.is_void), Decimal('0.00'))
      return float(_quantize_currency(total))
  ```

**What the audit misses:**
- The codebase has *separate* scoped methods — `Student.get_checking_balance(join_code=...)` and `Student.get_savings_balance(join_code=...)` at `app/models.py:314-409` — that properly filter by `join_code`.
- All critical financial operations (student dashboard, transfers, rent payments, payroll) use these scoped methods, **not** the unscoped properties.

**Where the vulnerability actually exists (3 admin display routes):**

| Route | File:Line | Usage |
|-------|-----------|-------|
| Admin dashboard stats | `app/routes/admin.py:733` | `sum(s.checking_balance + s.savings_balance ...)` |
| CSV export | `app/routes/admin.py:7090-7091` | `f"{student.checking_balance:.2f}"` |
| Banking page totals | `app/routes/admin.py:7655-7660` | `sum(s.checking_balance for s in students)` |

**Adjusted Severity:** Medium (data display issue, not a financial integrity breach). No student can spend cross-period funds because all transaction-creating code paths use the scoped methods.

### Finding 2: Void Logic Double Deduction

**Audit Claim:** When voiding a transaction, the code (1) sets `is_void = True` on the original and (2) creates a reversal transaction with negative amount. Since balance calculation sums where `is_void == False`, the original is excluded AND the reversal is included, causing a double deduction.

**Verdict: NOT CONFIRMED — Audit finding is incorrect**

The audit's logic contains a sign error. Here is what actually happens:

1. Original transaction (e.g., +$100 payroll deposit) is marked `is_void = True` → **excluded** from balance sum.
2. Reversal transaction is created with **inverted sign** (−$100) and `is_void = False` → **included** in balance sum.

Net effect on balance: the original +$100 disappears (excluded), and −$100 is added. Balance decreases by $100 total — exactly one deduction, not two.

**Code evidence (`app/routes/admin.py:6472-6490`):**
```python
transaction.is_void = True
reversal_amount = -(transaction.amount or Decimal('0.00'))
reversal_tx = Transaction(
    student_id=transaction.student_id,
    amount=reversal_amount,  # inverted sign
    ...
)
```

**Balance calculation (`app/models.py:314-375`):** Sums all transactions where `is_void == False`. The voided original is excluded; the reversal (with opposite sign) is included. The math is correct.

**Test confirmation:** `tests/test_banking_core.py:102-151` (`test_void_posted_with_reversal`) explicitly asserts correct post-void balance.

**Settlement engine (`app/utils/banking.py:158-176`):** Voided pending transactions are skipped entirely (`continue`), never contributing to the balance cache.

### Finding 3: Random Ledger Assignment in Bulk Bonuses

**Audit Claim:** `give_bonus_all` (`app/routes/admin.py:899-1014`) builds a `join_code_map = {student_id: join_code}` via dictionary comprehension. If a student is enrolled in multiple classes with the same teacher, earlier entries are overwritten, assigning the bonus to an arbitrary period.

**Verdict: CONFIRMED**

**Code evidence (`app/routes/admin.py:913-923`):**
```python
teacher_blocks = (
    TeacherBlock.query
    .filter(
        TeacherBlock.student_id.in_(sa.select(student_ids_subquery)),
        TeacherBlock.teacher_id == current_admin_id,
        TeacherBlock.is_claimed.is_(True)
    )
    .with_entities(TeacherBlock.student_id, TeacherBlock.join_code)
    .all()
)
join_code_map = {student_id: join_code for student_id, join_code in teacher_blocks}
```

The dictionary comprehension on line 923 uses `student_id` as key. For a student in both PERIOD1 and PERIOD2, whichever row is returned last wins. The bonus transaction is created with that single `join_code` (`admin.py:960`), meaning it lands in one arbitrary period's ledger.

The function does not use `session['current_join_code']` to scope the operation. The fallback on lines 936-941 only handles `None` join codes, not the overwrite scenario.

**Severity confirmed as High.** Multi-period enrollment with the same teacher is an explicitly supported scenario per `docs/security/CRITICAL_SAME_TEACHER_LEAK.md`.

### Finding 4: Float Usage in Financial Calculations

**Audit Claim:** `Student.checking_balance` returns `float`, `Transaction.amount` is `Decimal`, and `app/routes/api.py` compares floats against Decimals, risking precision errors.

**Verdict: PARTIALLY CONFIRMED (severity overstated)**

**What is true:**
- `Transaction.amount` is `db.Numeric(precision=12, scale=2)` → returns `Decimal` (`app/models.py:706`).
- `Student.checking_balance` property explicitly converts to `float` at `app/models.py:293`.
- `app/routes/api.py:456` compares the float property against price values.

**Why severity is overstated:**
- All values are quantized to 2 decimal places via `_quantize_currency()` **before** the float conversion. IEEE 754 double-precision can exactly represent all 2-decimal-place currency values within the range used by this application.
- The scoped balance methods (`get_checking_balance`, `get_savings_balance`) return `Decimal`, not `float`. Critical financial operations (transfers at `app/routes/student.py:1556`) use these Decimal-returning methods.
- The `BalanceCache` model (`app/models.py:763`) stores amounts as integer cents, avoiding floating-point entirely for cached ledger values.

**Adjusted Severity:** Low. The float conversion is intentional and documented in a code comment. Precision loss is theoretically possible but practically unreachable given the 2-decimal quantization before conversion.

### Finding 5: Hard Deletion of Financial Records

**Audit Claim:** `_hard_delete_join_code_scope` (`app/routes/admin.py:258-420`) executes `DELETE` statements on the transactions table when a class is deleted, destroying audit trail.

**Verdict: CONFIRMED as code behavior, but this is INTENDED DESIGN**

**Code evidence (`app/routes/admin.py:252-255`):**
```python
def _delete_transactions_for_join_code(join_code, *, join_code_deletion=False):
    """Hard-delete transactions scoped to join_code (join-code destruction only)."""
    _assert_transaction_deletion_allowed(join_code, join_code_deletion=join_code_deletion)
    return Transaction.query.filter_by(join_code=join_code).delete(synchronize_session=False)
```

Called at `app/routes/admin.py:336` within `_hard_delete_join_code_scope`.

**Additional context:**
- 23+ tables are hard-deleted when a join code is removed, including `Transaction`, `TapEvent`, `RentPayment`, `InsuranceClaim`, `StudentInsurance`, `AnalyticsSnapshot`, and more.
- Test at `tests/test_join_code_deletion_semantics.py:217-218` explicitly asserts permanent removal.
- Triggered via `/admin/join-code/delete` (line 2817) and `/admin/class/delete` (line 2779).

**Why this is not a defect:** The hard deletion is intentional, implementing a strict privacy and minimum data retention policy. When a teacher requests deletion of a class (scoped by `join_code`), all corresponding records are permanently removed. This is a deliberate design choice — the system does not retain student financial data beyond the lifetime of the class it belongs to. The audit frames this as an "audit trail violation," but the project prioritizes data minimization over long-term record retention.

**Severity: Not applicable (intended behavior).**

## Recommendations

### Immediate (based on confirmed findings)

1. **Fix bulk bonus join_code assignment** (Finding 3): Replace the dictionary comprehension with logic that either scopes to `session['current_join_code']` or creates one bonus per enrolled period.

2. **Replace unscoped balance properties** (Finding 1): Deprecate `Student.checking_balance` and `Student.savings_balance` properties, or require a `join_code` parameter. Update the 3 admin routes that use them.

### Medium-term

3. **Migrate to Decimal throughout** (Finding 4): Remove the `float()` conversion in balance properties. While current quantization makes this low-risk, using Decimal end-to-end is standard practice for financial systems.

### Not required

4. **Void logic** (Finding 2): No action needed. The current implementation is mathematically correct.

5. **Hard deletion of records** (Finding 5): No action needed. The hard deletion is intentional — the project follows a strict privacy and minimum data retention policy where class deletion removes all scoped records.
