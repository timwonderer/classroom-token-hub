# ARC-OPS-015: Multi-tenancy and Join-Code Interface

| Reference Number | Version | Effective Date | Supersedes | Authority Level |

## I. Purpose
Define the architectural relationship between public access identifiers (`join_code`), private tenant containers (`ClassEconomy`), and student identity bindings (`Seat`).

## II. Scope
Applies to all multi-tenant boundaries, student join/claim flows, and cross-class scoping enforcement logic.

## III. Authority Level
Constitutional (ARC Tier). Subordinate to INV-CORE-000 (Section 1 and 6).

## IV. Dependencies
- `INV-CORE-000_Core_Invariants.md`
- `ARC-IDEN-001_Admin_Identity_Handling.md`

## V. Tenant Identification Model

The system utilizes a dual-identifier strategy to balance public ease-of-use with private architectural stability.

### 1. The Public Pointer: `join_code`
- **Definition**: A short, human-friendly alphanumeric string (e.g., `ABC-123`).
- **Function**: Acts as a public routing key allowing students to locate a specific class environment.
- **Mutability**: The `join_code` can be regenerated by a teacher if the previous code is compromised, without breaking the underlying class identity.
- **Authority**: The `join_code` is a property of the `ClassEconomy` model.

### 2. The Private Authority: `class_id`
- **Definition**: A globally unique UUID4 identifier.
- **Function**: The primary database join key for all scoped data (Ledger, Analytics, Inventory).
- **Persistence**: Never changes for the life of the class instance.
- **Constraint**: Must never be exposed to student-facing public endpoints.

## VI. The Seat Identity Model

Identity within a specific `ClassEconomy` is established via the `Seat` abstraction, which separates a student's global identity from their class-specific state.

### 1. Identity Bridge: `ClassMembership`
- A student establishes a relationship with a class via a `ClassMembership` record.
- Access is gated by the presence of this record. Deletion of the membership record results in the immediate loss of all class-scoped access for that student.

### 2. Student Binding: `Seat`
- **Placeholder State**: A "Seat" (or `unclaimed` student) is created by the teacher. It possesses a balance and a name but lacks a bound user account.
- **Claim Flow**: A student user (System-wide `Student` record) uses a `join_code` to locate a class and "claims" an available seat. 
- **Mapping**: This binding persists as a `ClassMembership` entry linking `Student.id` to `ClassEconomy.id`.

## VII. Scoping Enforcement
All domain queries must enforce the following hierarchy:
1. Identify the target `class_id` from the active context or request parameter.
2. Verify the requesting user has an active `ClassMembership` for that specific `class_id`.
3. Filter all associated data (Transaction, HallPass, Announcements) by that `class_id`.

## VIII. Deletion Semantics
- **Tenant Deletion**: Deleting a `ClassEconomy` must trigger a cascading purge of all `ClassMembership` and `Seat` records for that class.
- **Membership Deletion**: Removing a student from a class deletes their membership record only. If the global `Student` record has zero remaining memberships, it must be purged (per INV-CORE-000 Section 6).
